  What the Backend Does

  LegalSignal's backend is an AI visibility monitoring pipeline for law firms. It answers: "When someone asks an AI chatbot for lawyer
   recommendations in Dallas, does your firm get mentioned?"

  The Pipeline (end-to-end in pipeline.py)

  1. QUERY    → Send 26 legal prompts to 3 AI platforms (Perplexity, ChatGPT, Gemini)
  2. EXTRACT  → Use GPT-4o-mini to pull structured firm mentions from each response
  3. RESOLVE  → Match extracted names to canonical firms via alias lookup + fuzzy matching
  4. SCORE    → Compute a 0-100 visibility score from mention rate, position, and sentiment
  5. REPORT   → Generate a PDF report + HTML email, send via Resend
  6. STORE    → Everything persisted to Supabase (runs, responses, scores)

  Each step is synchronous, single-process, no task queue. The whole run for one client takes ~2-5 minutes depending on how many
  prompts and platforms are configured.

  Gaps and Oversights

  Bugs

  1. Gemini retry_error_callback returns None (providers/gemini.py:23). When all 3 retries fail, query() returns None instead of a
  ProviderResult. The pipeline then calls result["raw_text"] on None and crashes. Should either raise an exception or return an empty
  ProviderResult.
  2. run_migrations() only prints 001_initial_schema.sql (db/connection.py:35). Migrations 002-004 are never shown. The function
  should iterate all .sql files in order.
  3. Per-platform score denominator is wrong (pipeline.py:259-261). platform_prompts = len(prompts) is correct, but the
  client_mentions list isn't filtered by platform before passing to compute_visibility_score. The function internally filters, but
  total_prompts should be len(prompts) (prompts per platform), which it is — this is actually fine. However, total_prompt_count on
  line 235 is len(prompts) * len(providers), which is the cross-product. The overall score uses this cross-product as denominator,
  meaning mention_rate is computed per prompt-platform pair, not per prompt. This is intentional but may not match client
  expectations.
  4. normalize_firm_name doesn't strip "&" despite the root CLAUDE.md claiming it does. The function only strips suffixes; standalone
  & and "and" in the middle of names are preserved. This is correct behavior for matching but inconsistent with the documentation.

  Missing Functionality

  5. No admin review workflow in the backend. Resolution flags firms as needs_review: True, but there's no endpoint or CLI command to
  list/resolve review items. The frontend has an admin review page, but it hits a route (/api/admin/review/resolve) that writes
  directly to Supabase — there's no backend validation.
  6. No scheduled execution. Railway runs the pipeline on deploy (startCommand), but there's no cron job or scheduler. Weekly runs
  require manual deployment or an external trigger.
  7. No historical trend tracking. visibility_scores stores one row per client per week, but there's no logic to compute
  week-over-week trends, moving averages, or alerts on score drops.
  8. No prompt rotation or A/B testing. All 26 prompts run every time. No mechanism to test new prompts, retire stale ones, or vary
  prompts per run.
  9. No rate limit handling beyond retries. If a provider rate-limits consistently (429s), the pipeline just retries 3 times and moves
   on. No backoff across the full run, no circuit breaker.

  Architectural Concerns

  10. seed_registry uses insert not upsert (db/connection.py:86). Unlike seed_prompts which upserts, registry seeding catches
  duplicate errors in a try/except. This means re-running seed-registry won't update existing firms — only skip them.
  11. No input validation on geo_config. Pipeline passes client.get("geo_config") or {} directly to providers. If geo_config is
  malformed (e.g., missing city), providers use hardcoded Dallas defaults, which silently produces wrong results for non-Dallas
  clients.
  12. Competitor scores are computed relative to the client's prompts, not the competitor's own relevant prompts. A family law firm's
  competitors get scored on PI prompts if the client is PI-focused. This is actually correct (measuring competitor visibility in your
  space) but should be documented.
  13. No logging configuration beyond basicConfig. All modules use logging.getLogger(__name__) but there's no structured logging, no
  log levels per module, and no JSON formatting for production.
